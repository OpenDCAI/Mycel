Architecture Research

Thread 级定时任务系统

目标不是“定时创建一条任务”，而是“定时唤醒某个长期存在的 brain thread，让它执行一项工作，同时和任务系统保持同一个语义世界，对外有清晰边界”。

现状调研 方案比较 推荐模型 边界设计 实施分期

现状结论

已有的 Scheduler

现在的 CronService 只会扫描 cron_jobs，然后往 panel_tasks 里插一条记录。它不会驱动 thread，也不会保证 agent 被唤醒。

已有的 Thread 唤醒链路

route_message_to_brain() 已经能把消息投递到长期存在的 thread：如果 thread 空闲就直接启动 run，如果忙就入 followup queue。

已有的 Task 语义过多

仓库里至少有三套任务：panel_tasks、core/tools/task 里的 thread todo、还有 /threads/{id}/tasks 这套后台 run 查询接口。名字相同，语义不同。

关键问题

如果继续在现有 cron_jobs -> panel_tasks 上补丁式迭代，最后会得到一个“定时、任务、后台 run、thread 注入”互相缠绕的系统。功能能堆出来，但边界会继续塌。

当前系统其实长这样

今天的定时链路

cron_jobs 定时定义：cron 表达式 + task_template JSON
↓ every 60s
CronService 判断是否 due，创建 panel task
↓
panel_tasks UI 可见任务板记录，source=cron

今天的 thread 唤醒链路

route_message_to_brain() 接收一段消息
↓
IDLE 直接 start_agent_run()
or
ACTIVE 入 queue，等当前 run 结束后继续
事实判断

现成能力已经足够实现“定时直驱 thread”，但当前代码没有把 scheduler 和 thread delivery 收进同一个任务域模型里。

三种候选架构

方案 核心做法 优点 问题 结论
A. 在现有 cron 上硬接 thread 给 cron_jobs 增加 thread_id，到点后直接 route_message_to_brain()，顺手写一条 panel_tasks。 改动最少，最快落地。 还是把“调度定义”“任务实例”“执行结果”揉在一起，长期会继续脏。 不推荐
B. 建一个统一的 Tasks 域模块 把 schedule、task definition、task run、thread dispatch 收到一个模块里；UI 上的任务板只是它的 projection。 边界清楚，能把“定时”和“任务”变成同一体系。 需要做一次结构调整，但范围可控。 推荐
C. 再加一层事件总线 / outbox scheduler 只产生命令，dispatcher 异步消费再投递 thread。 扩展性、恢复性最好。 当前仓库还没到这个复杂度，容易过度设计。 保留为第二阶段

推荐建模

推荐句子版

把“定时任务”定义成一种 Task Trigger，它触发的是同一个任务域里的 Task Run；Task Run 的执行目标是长期存在的 thread；对外暴露的只有任务 API 和调度 API，内部再通过 thread delivery gateway 去唤醒 brain。

建议的新域对象

  • TaskSpec：任务定义。描述要做什么，绑定哪个 thread，注入什么 prompt/template。
  • TaskTrigger：触发规则。cron 只是其中一种 trigger。
  • TaskRun：某次真正执行实例。状态、开始时间、结束时间、失败原因、关联 run_id 都在这里。
  • ThreadDispatchGateway：唯一负责把 TaskRun 投递进 thread 的边界口。

明确不要再混在一起的东西

  • 不要再让 task_template 继续做裸 JSON 黑箱。
  • 不要让 cron_jobs 直接变成“任务实例表”。
  • 不要让前端任务板记录直接代表执行状态真相。
  • 不要让 scheduler 直接知道 UI 细节。

建议的模块边界

backend/tasks/ 任务域主模块。只关心 TaskSpec / Trigger / Run / Dispatch。
↓ expose ports
infra repositories SQLite / storage provider 实现：task_specs、task_triggers、task_runs。
thread gateway 通过现有 route_message_to_brain() 和 start_agent_run() 投递。
↓ projections
panel projection UI 任务板读取 task_runs 或 task_specs 的投影，而不是自己独立长一套语义。
scheduler worker 只负责 claim due trigger → create run → dispatch，不直接做业务分支判断。
外部边界应该很明显

外界不应该直接操作 thread queue、background run registry、panel_tasks 这些内部实现。外界应该只看到：创建任务定义、配置调度、查看执行实例、取消执行实例。

推荐的数据关系

task_specs

定义层。字段建议包含：id、thread_id、title、instruction_template、enabled、created_at。

task_triggers

触发层。字段建议包含：id、task_spec_id、type=cron、cron_expression、next_fire_at、last_fire_at。

task_runs

实例层。字段建议包含：id、task_spec_id、trigger_id、thread_id、status、dispatch_run_id、started_at、completed_at、result_summary。

执行流应该变成什么样

1. Scheduler 扫描 due trigger，claim 一条 trigger
↓
2. TaskRunFactory 创建一条 task_run，状态 = dispatching
↓
3. ThreadDispatchGateway 生成调度消息，调用 route_message_to_brain(thread_id, ...)
↓
4. Thread 空闲则立即执行，忙则入 queue
↓
5. Run Correlator 把 thread run_id / 完成状态 / 失败原因写回 task_run

为什么这比现在好

统一语义

“定时任务”和“任务实例”不再平行。定时只是 trigger，真正的执行单位始终是 task_run。

thread 是一等公民

执行目标是长期存在的 brain，而不是临时 agent。符合你的核心要求。

边界清楚

外界调 API 看任务；内部如何投递到 queue / start run / correlate 完全被关在 tasks 域内。

前端也会更干净

任务板看的是 task_run 投影，调度页看的是 trigger 配置，不再混着展示假字段。

实施计划草案

Phase 1. 语义收口

  1. 引入 backend/tasks/ 模块，定义 TaskSpec / Trigger / Run。
  2. 保留旧 API 入口，但改成调用新模块。
  3. 把现有 cron_jobs 和 panel_tasks 明确标记成 legacy projection。

Phase 2. Thread 调度接入

  1. 增加 thread-bound task spec。
  2. scheduler 到点后创建 task_run，不再只创建 panel task。
  3. 通过 thread gateway 调 route_message_to_brain() 触发执行。

Phase 3. 状态对齐

  1. 把 thread run_id 和 task_run 建立关联。
  2. 完成 / 失败 / 取消回写 task_run。
  3. 前端任务板切到 task_run projection。

Phase 4. 清理旧结构

  1. 裁剪旧 cron_service、旧 cron UI 假字段。
  2. 决定 core/tools/task 是否并入同一任务域，还是保留为 thread-local checklist 工具。
  3. 必要时再引入 outbox / worker 分离。

唯一还没完全定死的问题
